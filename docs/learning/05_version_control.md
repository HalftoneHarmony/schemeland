# 5. 버전 컨트롤 (Version Control - Git & GitHub)

### Q: Antigravity가 버전 관리를 알아서 해주나요?
**A: 반은 맞고 반은 틀립니다.**
Antigravity는 전 세계 표준인 **Git(깃)**을 사용합니다. 제가 내부적으로 독자적인 시스템을 쓰는 게 아니라, 개발자들이 쓰는 **진짜 Git 명령어**를 대신 입력해 드리는 것입니다.

지금 제가 프로젝트 폴더에서 `git init`, `git commit`을 해서 로컬(내 컴퓨터)에 저장소는 만들었습니다. 하지만 **GitHub(인터넷 저장소)**에 올리는 건 사용자의 계정 권한이 필요해서, 최초 1회 연결 작업이 필요합니다.

### 🚀 GitHub에 내 프로젝트 올리는 법 (따라해 보세요!)

1.  **GitHub 저장소 만들기**
    *   [GitHub.com](https://github.com)에 로그인합니다.
    *   우측 상단 `+` 버튼 -> `New repository` 클릭.
    *   **Repository name**: `schemeland` (또는 원하는 이름)
    *   Public/Private 선택 후 `Create repository` 클릭.

2.  **연결 주소 복사하기**
    *   생성된 화면에서 `HTTPS` 주소를 복사합니다. 
    *   예: `https://github.com/사용자ID/schemeland.git`

3.  **Antigravity에게 주소 알려주기**
    *   채팅창에 이렇게 말해주세요:
    *   `"깃허브 주소는 https://github.com/사용자ID/schemeland.git 야. 여기로 푸시해줘."`

### 💡 용어 정리
*   **Git (깃)**: 내 컴퓨터에서 파일의 역사를 기록하는 타임머신.
*   **GitHub (깃허브)**: 그 기록을 안전하게 보관하는 구글 드라이브 같은 클라우드.
*   **Commit (커밋)**: "이 시점으로 저장해!" 하고 사진 찍는 것 (세이브 포인트).
*   **Push (푸시)**: 내 컴퓨터의 세이브 파일을 깃허브로 업로드하는 것.

---

### 📅 언제 Git을 써야 하나요? (Good Habits)

**Q: "얼마나 자주 저장(Commit)해야 돼?"**

#### 1. "의미 있는 단위"로 완성했을 때 (Save Game)
게임을 하다가 보스를 깼거나, 중요한 아이템을 얻으면 바로 세이브를 하죠? 코딩도 똑같습니다.
*   ✅ "로그인 기능 완성했어!" -> **Commit**
*   ✅ "버튼 디자인 예쁘게 바꿨어!" -> **Commit**
*   ✅ "오타 수정했어." -> **Commit**

#### 2. "불안할 때" (Checkpoint)
"이거 건드리면 다 망가질 것 같은데...?" 하는 어려운 작업을 시작하기 전이나, 작업 도중 잘 돌아가는 상태일 때 일단 저장하세요. 망하면 바로 돌아올 수 있습니다.

#### 3. 하루를 마무리는 때 (Sync)
작업을 끝내고 컴퓨터를 끄기 전에는 꼭 **Commit & Push**를 하는 습관을 들이세요.
*   내 컴퓨터가 갑자기 고장 나도 깃허브에 코드가 살아있습니다.
*   다음 날 다른 컴퓨터에서도 이어서 작업할 수 있습니다.

### ✨ 이것만은 지켜주세요 (Best Practices)
1.  **커밋 메시지는 솔직하고 명확하게**: "점심 먹고 수정" (X) -> "메인 화면 로고 크기 변경" (O)
2.  **에러가 있는 채로 커밋하지 않기**: 세이브 파일을 로드했는데 게임이 켜지자마자 꺼지면 안 되겠죠? 실행은 되는 상태에서 저장하는 게 좋습니다.

### 자주 묻는 질문 (FAQ)

### Q. 커밋(Commit)만 해도 예전으로 돌아갈 수 있나요? (Push/Pull 없이?)
**네, 가능합니다!** ✅

*   **Commit(커밋)**은 내 컴퓨터(로컬)에 저장하는 행위입니다. 인터넷 연결이 없어도, GitHub에 올리지 않아도 커밋만 해뒀다면 언제든 그 시점으로 시간 여행(Reset/Checkout)을 할 수 있습니다. (마치 게임의 세이브 포인트처럼요!)
*   **Pull(풀)**은 남이 작업한 걸 내 컴퓨터로 가져오는 것입니다. 혼자 작업할 때는 내 과거로 돌아가는 것과 관계가 없습니다.
*   **Push(푸시)**는 내 컴퓨터가 고장 날 때를 대비해 구글 드라이브(GitHub)에 업로드하는 것입니다.

**결론**: 내 컴퓨터 안에서 시간을 되돌리는 건 **커밋(Commit)**만 잘 되어 있으면 언제든 가능합니다! 하지만 컴퓨터가 고장 나는 것을 대비해 **Push**도 자주 해주면 더 안전하겠죠.

## 🌟 고급: 여러 명이 동시에 일하는 법 (브랜치 워크플로우)

비개발자 사용자가 여러 AI 에이전트와 협업할 때 가장 강력한 도구는 바로 **Branch(브랜치)**입니다.

### 1. Branch(브랜치)란? = "평행 세계"
프로젝트의 메인 줄기(`main`)에서 줄기를 하나 따서 별도의 작업 공간을 만드는 것입니다. 
- **Main(메인)**: 언제든 손님에게 보여줄 수 있는 '완성된 요리'
- **Branch(브랜치)**: 주방 뒤편에서 새로운 레시피를 실험하는 '개인 조리대'

### 2. 왜 브랜치부터 만들고 시작해야 하나요?
1. **안전성**: 에이전트가 코드를 엉망으로 짜도 메인 화면은 멀쩡합니다. 맘에 들 때만 합치면 되니까요.
2. **독립성**: 디자인 에이전트와 데이터 에이전트가 서로의 발을 밟지 않고 각자 자기 조리대에서 작업할 수 있습니다.
3. **깔끔한 기록**: 어떤 작업을 위해 어떤 파일을 고쳤는지 브랜치 이름만 봐도 알 수 있습니다.

### 🤖 AI 에이전트에게 일 시키는 법 (Best Practice)
다음에 새로운 기능을 요청할 때는 이렇게 말해보세요:

> **"지금부터 [기능 이름] 작업을 할 거야. 우선 `feat/[기능이름]` 브랜치를 먼저 만들고, 거기서 작업을 시작해줘. 작업이 끝나면 커밋하고 알려줘."**

### 🛠️ 협업 프로세스 (순서)
1. **브랜치 생성**: 새로운 평행 세계를 만듭니다.
2. **작업 및 커밋**: 해당 브랜치에서 한 발자국씩 저장합니다.
3. **검토**: 에이전트가 작업한 결과물이 맘에 드는지 확인합니다.
4. **합치기(Merge)**: 맘에 든다면 메인 줄기(`main`)로 합칩니다.

이렇게 브랜치를 활용하면, 여러 에이전트와 복잡한 작업을 동시에 진행해도 프로젝트가 엉키지 않고 깔끔하게 유지됩니다!
