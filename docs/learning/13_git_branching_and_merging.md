# 🌳 버전 관리의 흐름: 브랜치(Branch)와 머지(Merge) 이해하기

비개발자가 협업 과정에서 가장 궁금해하는 **Git 워크플로우**에 대한 문답 정리입니다.

---

## 🙋 실시간 질문: "브랜치에서 작업해도 작동하는데, 왜 굳이 메인에 합쳐야 하나요?"

### 1. 브랜치(Branch)는 '안전한 실험실'입니다
- **질문**: 메인에 안 합쳐도 프리뷰에서 작동하는 걸 볼 수 있는데, 왜 합치나요?
- **답변**: 네, 브랜치를 바꾸면 코드 파일 자체가 교체되기 때문에 프리뷰에서도 바로 바뀐 내용을 볼 수 있습니다. 하지만 브랜치는 **'임시 작업 공간'**입니다. 
- **비유**: 갤럭시 폰의 새로운 기능을 개발자가 자기 폰(브랜치)에서 테스트하는 것과 같습니다. 개발자 폰에서는 잘 돌아가지만, 전체 사용자에게 배포하려면 **'공식 업데이트 서버(메인)'**에 합쳐야 합니다.

### 2. 왜 메인(Main)에 합쳐야(Merge) 할까요?
1. **기준점 세우기 (Single Source of Truth)**: 
   - 메인 브랜치는 프로젝트의 '공식적인 현재 상태'입니다. 
   - 합치지 않고 브랜치를 여러 개 만들면, 나중에 "A 기능과 B 기능을 합친 버전"을 찾을 수 없게 됩니다.
2. **협업의 약속**:
   - 다른 개발자나 AI가 새로 투입되었을 때, 항상 `main` 브랜치를 기준으로 작업을 시작합니다. `main`에 코드가 없으면 다른 사람은 당신이 만든 기능을 모른 채 옛날 버전으로 작업하게 됩니다.
3. **배포의 자동화**:
   - 실제 서비스(Website)는 보통 `main` 브랜치에 있는 코드만 자동으로 읽어서 세상에 공개하도록 설정되어 있습니다.

---

## 💡 개발 가이드: 언제 합치고 언제 나누나요?

- **나눌 때 (Branch Create)**: 새로운 기능(예: 유효성 검사, 디자인 수정)을 시작할 때. 기존 코드를 망칠 걱정 없이 자유롭게 실험하세요!
- **합칠 때 (Merge)**: 실험이 성공적으로 끝나고, "이제 이 기능은 우리 앱의 공식 기능이야!"라고 선언하고 싶을 때.
- **삭제할 때 (Branch Delete)**: 메인에 합치고 나면, 그 실험실(브랜치)은 더 이상 필요 없으므로 깔끔하게 정리합니다.
